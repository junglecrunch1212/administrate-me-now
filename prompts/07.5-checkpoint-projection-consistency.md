## The audit

Six sections. Each is a **finding statement** (PASS / FLAG / FAIL) followed by **evidence** (file paths, citations, test names). The audit memo is the Partner's writeup of all six.

### A. Per-projection consistency (11 projections)

For each of: `parties`, `interactions`, `artifacts`, `commitments`, `tasks`, `recurrences`, `calendars`, `places_assets_accounts`, `money`, `vector_search`, `xlsx_workbooks`:

1. **Schema present and self-consistent.** `schema.sql` exists; `CHECK` constraints on closed-enum columns (kind, status, sensitivity) per CF-6 (which graduated into preamble); `sensitivity TEXT NOT NULL DEFAULT 'normal' CHECK (sensitivity IN ('normal','sensitive','privileged'))` present. Composite PK `(tenant_id, <entity_id>)`.

2. **Handlers consume what's registered.** `handlers.py`'s `apply_event` dispatch table lists event types; every type appears in `subscribes_to` of the projection class (`__init__.py`). Cross-check against the schema registry: every subscribed type has a registered Pydantic schema at v1 (per D7).

3. **Cross-DB FK references are documentation-only.** Every comment-style reference (`-- REFERENCES parties(party_id)`) is a docstring annotation, not an enforced FK. `[§2.3]` says SQLite cannot enforce FKs across separate projection DBs; integrity is a pipeline-level concern.

4. **Privileged filter at handler time** for projections that touch sensitive data: `vector_search` drops privileged events (per `[§13.8]`). For projections that store privileged-capable rows (interactions, money, etc.), the row's `sensitivity` column carries the envelope value forward.

5. **TODO(prompt-08) markers** on query functions where Session is required. Count per projection. Total goes into the audit memo.

**Expected outcome:** PASS for all 11 unless something has drifted. Flag any inconsistency.

### B. xlsx_workbooks-as-projection + reverse daemon — the L3 / L1-adjacent pair

This is the audit scope that grew between draft and refactor.

1. **Forward xlsx is structurally a projection** (`adminme/projections/xlsx_workbooks/__init__.py`): subscribes to event types from 7 projections, regenerates two .xlsx files plus their per-sheet sidecar. Emits exactly one event type, `xlsx.regenerated`, classified as a system event per `[§2.2]` resolution. PASS by canary: `tests/integration/test_xlsx_forward_end_to_end.py` asserts no other type emits.

2. **Reverse xlsx daemon is L1-adjacent** (`adminme/daemons/xlsx_sync/reverse.py`): emits domain events on principal authority + two system events. Lives outside `adminme/projections/`, so the §2.2 projection-emit allowlist does not apply per PM-14. PASS by canary: `scripts/verify_invariants.sh` `ALLOWED_EMIT_FILES` excludes `daemons/`.

3. **Two forward-only sheet groups** per `_BIDIRECTIONAL_SHEETS` and `_READONLY_SHEETS` in the forward daemon. Bidirectional set: Tasks, Recurrences, Commitments (ops); Raw Data (finance). Read-only set: People, Metadata (ops); Accounts, Metadata (finance). Confirmed by descriptor coverage in `sheet_schemas.py` (one descriptor per bidirectional sheet, no descriptors for read-only sheets).

4. **Sidecar pathway closed at both ends.** Forward writes per-sheet sidecar inside the workbook lock (07c-α). Reverse rewrites at end of every cycle including no-op cycles (07c-β). PASS by canary: `test_xlsx_forward_writes_sidecar.py` (forward), `test_xlsx_reverse_cold_start.py` (reverse cold-start sidecar regeneration), `test_xlsx_roundtrip.py` (both halves end-to-end).

### C. DERIVED_COLUMNS ↔ always_derived equivalence — the named canary

Per `[BUILD.md §3.11]`: a forward sheet builder's `DERIVED_COLUMNS` set and the reverse descriptor's `always_derived` set must match exactly per (workbook, sheet). If they drift, the principal can edit a backend-derived column and the reverse daemon will emit a spurious `*.updated` event for it (or worse, treat the principal's edit as a deletion).

The audit walks each bidirectional sheet:

| Workbook | Sheet | Builder file | DERIVED_COLUMNS | Descriptor always_derived | Match |
|---|---|---|---|---|---|
| ops | Tasks | `sheets/tasks.py` | `{task_id, created_at, completed_at}` | `{task_id, created_at, completed_at}` | ✓/✗ |
| ops | Commitments | `sheets/commitments.py` | `{confidence, strength, source_summary}` | `{confidence, strength, source_summary}` | ✓/✗ |
| ops | Recurrences | `sheets/recurrences.py` | `{next_due, last_completed_at}` | `{next_due, last_completed_at}` | ✓/✗ |
| finance | Raw Data | `sheets/raw_data.py` | `ALWAYS_DERIVED = {txn_id, plaid_category}` | `{txn_id, plaid_category, is_manual}` | **MISMATCH expected** |

The Raw Data row is the live audit finding. The descriptor declares `is_manual` as always-derived (correctly — it's set by the projection from event type, not the principal); the sheet builder's `ALWAYS_DERIVED` set in `sheets/raw_data.py` does not include it, leaving cell-level protection off for `is_manual` even though no edit path acts on it.

**Concrete recommendation:** add `"is_manual"` to `ALWAYS_DERIVED` in `adminme/projections/xlsx_workbooks/sheets/raw_data.py`. One-line fix; defer to a sidecar PR named `sidecar-raw-data-is-manual-derived` per `E-session-protocol.md` "Single-purpose PR rule." Worth a unit test in the sidecar that asserts the cell-level protection on `is_manual` matches the descriptor.

The integration round-trip test (`test_xlsx_roundtrip.py`) is the **runtime canary** for any future drift of this kind: if a builder adds a derived column without updating the descriptor, the test will start emitting spurious `*.updated` events on the next change to the involved column and fail.
