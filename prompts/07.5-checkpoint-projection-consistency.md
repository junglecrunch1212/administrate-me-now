**Phase:** Phase A checkpoint. After prompt 07, before prompt 08.

## Read first (required)

1. `docs/SYSTEM_INVARIANTS.md` — Section 1 (event log), Section 2 (projections are derived), Section 3 (CRM spine), Section 4 (domain spine), Section 5 (calendar relationships).
2. All 11 projections in `adminme/projections/` — read each `schema.sql` and `handlers.py`.

## Operating context

Eleven projections now exist. Each was written independently over prompts 05, 06, 07. This checkpoint verifies they are internally consistent before later prompts (L4 pipelines, L5 surfaces) start querying them. Specifically, it checks that:

- Foreign-key-like references between projections resolve (a commitment's `owed_to_party_id` exists in parties projection — at least in test data).
- Shared concepts are represented consistently (a `member_id` means the same thing in tasks as in calendars).
- Event types consumed across projections don't have conflicting handler logic (two projections reading the same event interpret its fields the same way).
- Schema columns that should exist per BUILD.md are all present.
- Every projection's `rebuild()` produces the same state as its live state (per diagnostic d03).

If this checkpoint finds a contradiction, it's cheaper to fix now than after pipelines depend on the wrong interpretation.

## Objective

Produce `docs/checkpoint-07.5-report.md`. For each check below, the report shows PASS or FAIL with details.

## Out of scope

- Do NOT rewrite any projection in this checkpoint. If you find a bug, record it in the report and either fix it in a follow-up Claude Code session (operator decision) or note it as "deferred to prompt XX."
- Do NOT add new event types or schemas.

## Deliverables

### The checks

For each projection:
1. **Schema vs. BUILD.md:** does every column BUILD.md says should exist actually exist? Does every column that exists appear in BUILD.md?
2. **Event consumption list:** which events does this projection's handlers.py subscribe to? Cross-reference against the schema registry (every subscribed event type has a registered schema).
3. **Rebuild determinism:** run `projection.rebuild()` on a fresh DB against 100 fixture events; assert byte-identical to live state.

For cross-projection consistency:
4. **Shared ID consistency:** `party_id` referenced in commitments must resolve to parties. `member_id` in tasks resolves to parties with a membership record. `calendar_event_id` referenced anywhere resolves to calendars.
5. **Event-type disjunction:** are any two projections' handlers consuming the same event type? If yes, are they interpreting it consistently (same field meanings)?
6. **Query namespace:** are there any projection query functions with the same name in different projections? (Not a bug by itself, but worth knowing.)
7. **Session acceptance:** every projection query function accepts a `Session` parameter (from prompt 08). This checkpoint runs BEFORE prompt 08; flag this as "expected to be fixed in prompt 08" rather than a real issue.
